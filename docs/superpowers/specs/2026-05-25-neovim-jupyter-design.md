# nvim-jupyter Design Spec

**Date:** 2026-05-25  
**Repo:** `/home/robertcowher/rustprojects/neovim-jupyter`  
**Status:** Approved for implementation

---

## Overview

A fully functional Jupyter notebook manager for Neovim. Rust binary handles kernel lifecycle and the Jupyter ZMQ wire protocol. Lua plugin handles Neovim buffer management, `.ipynb` parse/serialize, extmarks, and keymaps. Same architectural pattern as `nvim-gfx` (sibling project at `/home/robertcowher/rustprojects/neovim-graphics-viewer`).

**V1 scope:** Inline cell view, text-only output, kernel management, cell execution, notebook save.  
**Out of V1 scope:** Image/rich output (V2 — will integrate with nvim-gfx), HTML rendering.

---

## Architecture

```
Neovim (Lua plugin)
    ↕  JSON over stdin/stdout  (vim.fn.jobstart / chansend)
nvim-jupyter  (Rust daemon — one per Neovim session)
    ↕  Jupyter wire protocol over ZMQ
Jupyter kernels  (python3, julia, etc. — one per open notebook)
```

**Lua owns:**
- `.ipynb` parse and serialize (`vim.json.decode` / `vim.json.encode`)
- Scratch buffer creation and population
- Extmarks: cell boundary separators (`ns_cells`) and output virtual text (`ns_output`)
- Kernel state machine per buffer (`idle → starting → ready → busy → idle`)
- Keymaps and user commands
- Kernel picker UI (`vim.ui.select`)
- `BufReadCmd` / `BufWriteCmd` autocmds for `.ipynb` files

**Rust owns:**
- Kernel process lifecycle (spawn, kill, restart, interrupt)
- ZMQ socket management (shell, iopub, control, heartbeat — one set per kernel)
- Jupyter wire protocol: message framing, HMAC-SHA256 signing, message type structs
- Kernel discovery via `jupyter kernelspec list --json`
- Routing commands from Lua to the correct kernel task (by `kernel_id`)
- Forwarding kernel output events to Lua over stdout in real time
- Heartbeat monitoring (3 missed pongs → `kernel_died` event)

**Key constraint:** Rust uses `tokio` + the `zeromq` crate (pure-Rust ZMQ — no `libzmq` system dependency required).

## Rust File Layout

```
src/
  main.rs      — tokio::main, stdin reader task, router task, stdout writer
  protocol.rs  — Command and Event serde types (all IPC message shapes)
  router.rs    — HashMap<kernel_id, KernelHandle>; dispatch commands; forward events to stdout
  kernel.rs    — spawn kernel process; generate connection file; kill/interrupt
  client.rs    — ZMQ socket set per kernel (shell, iopub, control, heartbeat)
  wire.rs      — Jupyter wire protocol: message framing, HMAC-SHA256 signing, message type structs
```

### Cargo.toml dependencies

```toml
[dependencies]
serde       = { version = "1", features = ["derive"] }
serde_json  = "1"
tokio       = { version = "1", features = ["full"] }
zeromq      = "0.4"        # pure-Rust ZMQ, no libzmq needed
uuid        = { version = "1", features = ["v4"] }
hmac        = "0.12"
sha2        = "0.10"
```

### Runtime model

Three classes of tokio tasks:

1. **Stdin task** — reads lines from stdin, deserializes to `Command`, sends to router via `mpsc` channel.
2. **Router task** — owns `HashMap<String, KernelHandle>`, receives commands, spawns/kills kernel tasks, collects `Event`s from all kernel tasks and writes them to stdout as JSON lines.
3. **Kernel task** (one per live kernel) — owns ZMQ socket set; runs execute/iopub loop; sends `Event`s back to router via `mpsc` channel. Exits cleanly when kernel process dies or `stop_kernel` is received.

The Rust binary is launched once by Lua when the first notebook opens and stays alive for the Neovim session. It exits when Lua sends `{"cmd":"quit"}` or when its stdin closes (Neovim exits).

## Protocol

All messages are newline-delimited JSON. One object per line.

### Lua → Rust (commands, written to Rust stdin)

```jsonc
// Kernel lifecycle
{"cmd": "start_kernel",     "kernel_id": "<uuid4>", "kernel_name": "python3", "cwd": "/path/to/notebook/dir"}
{"cmd": "stop_kernel",      "kernel_id": "<uuid4>"}
{"cmd": "restart_kernel",   "kernel_id": "<uuid4>"}
{"cmd": "interrupt_kernel", "kernel_id": "<uuid4>"}

// Execution
{"cmd": "execute", "kernel_id": "<uuid4>", "msg_id": "<uuid4>", "code": "df.head()"}

// Discovery
{"cmd": "list_kernels"}

// Session end
{"cmd": "quit"}
```

`kernel_id` is generated by Lua using `vim.fn.system("uuidgen"):gsub("%s+", "")` and assigned at `start_kernel`. `msg_id` is generated the same way per execution to correlate output events back to the correct cell.

### Rust → Lua (events, written to Rust stdout)

```jsonc
// Kernel lifecycle
{"event": "kernel_started", "kernel_id": "<uuid4>"}
{"event": "kernel_ready",   "kernel_id": "<uuid4>"}
{"event": "kernel_died",    "kernel_id": "<uuid4>", "code": 1}

// Discovery
{"event": "kernels_list", "kernels": [{"name": "python3", "display_name": "Python 3", "language": "python"}]}

// Execution output (arrive before execute_done, in order)
{"event": "stream",          "kernel_id": "<uuid4>", "msg_id": "<uuid4>", "name": "stdout", "text": "hello\n"}
{"event": "execute_result",  "kernel_id": "<uuid4>", "msg_id": "<uuid4>", "execution_count": 42, "text": "   col1\n0     1"}
// execute_result.text is extracted by Rust from data["text/plain"] in the Jupyter execute_result or display_data message.
// All other mime types (text/html, image/png, etc.) are ignored in V1.
{"event": "execute_error",   "kernel_id": "<uuid4>", "msg_id": "<uuid4>", "ename": "ValueError", "evalue": "bad value", "traceback": ["line1", "line2"]}

// Execution complete
{"event": "execute_done", "kernel_id": "<uuid4>", "msg_id": "<uuid4>", "status": "ok"}
// status is "ok" or "error"

// Daemon-level error
{"event": "error", "msg": "human-readable description"}
```

## Lua File Layout

```
lua/nvim-jupyter/
  init.lua      — setup(); registers user commands; registers BufReadCmd/BufWriteCmd autocmds for *.ipynb
  daemon.lua    — Rust process lifecycle: jobstart, chansend, on_stdout event dispatch, on_exit handler
  notebook.lua  — .ipynb parse (vim.json.decode) → cell list; cell list → .ipynb JSON serialize
  cells.lua     — ns_cells extmark namespace; cell CRUD (add/delete/move); current-cell detection by cursor line
  output.lua    — ns_output extmark namespace; clear/set virtual text per cell; truncation at max_output_lines
  kernels.lua   — kernel_id generation (uuid4 in Lua); kernel state per buffer; vim.ui.select picker
  config.lua    — user-facing defaults: keymaps, highlight groups, virtual text prefix, max_output_lines
```

### Buffer lifecycle

**Open:**
1. `BufReadCmd *.ipynb` fires — Lua intercepts before Neovim reads the raw JSON.
2. `notebook.load(path)` reads file with `vim.fn.readfile`, decodes with `vim.json.decode`.
3. Creates a scratch buffer (`nvim_create_buf(false, true)`), sets `buftype=acwrite`, `filetype=python`. The buffer name is set to the `.ipynb` path so `BufWriteCmd` fires correctly on `:w`.
4. Populates buffer lines from `cells[*].source` (concatenated, one cell after another). All cells share the `python` filetype; markdown cells rely on TreeSitter injection syntax highlighting if available, otherwise display as plain text.
5. `cells.init(bufnr, cell_list)` draws `ns_cells` extmarks at each cell boundary.
6. Triggers kernel auto-start (see Kernel Startup Flow below).

**Save:**
1. `BufWriteCmd` fires on `:w`.
2. Lua reads current buffer lines.
3. Walks `ns_cells` extmarks (sorted by line) to reconstruct cell boundaries.
4. For each cell: slices lines between adjacent marks, trims trailing blank lines, preserves original cell metadata (outputs from last execution, `cell_type`, `execution_count`, `id`).
5. Encodes full notebook back to `.ipynb` JSON with `vim.json.encode`, writes with `vim.fn.writefile`.
6. Clears modified flag: `vim.bo[bufnr].modified = false`.
7. If two marks land on the same line (user deleted a boundary), merge those cells with a `vim.notify` warning.

### Extmark strategy

Two namespaces per buffer:

**`ns_cells`** — one `nvim_buf_set_extmark` per cell boundary. Uses `virt_lines` to draw the separator *above* the first line of each cell:
```
─── code ─────────────────────────────────────────
```
Extmarks track through buffer edits automatically. Cell type (code/markdown) and original `.ipynb` cell metadata (id, outputs, execution_count) are stored in a module-level Lua table in `cells.lua` keyed by extmark ID: `M._cell_meta[extmark_id] = {cell_type="code", id="...", outputs={}, execution_count=nil}`. Do not use the `user_data` field on the extmark itself — it is unreliable across buffer reloads.

**`ns_output`** — one extmark per cell, placed at the cell's *last line*, using `virt_lines_above = false` (drawn below). Contains output virtual text lines. Cleared and rewritten on each execution. Prefix each line with a dimmed `▷ ` glyph. Truncate at `config.max_output_lines` with a `[... N more lines]` indicator.

### Kernel state machine (per buffer)

```
idle → starting → ready → busy → idle
                         ↓ (on error/kernel_died)
                        dead
```

Stored in `vim.b[bufnr].jupyter_kernel_status` (string). Statusline can read this directly.

### Highlight groups

Defined in `config.lua`, overridable by user colorscheme:
```
NvimJupyterCellSep     — cell separator line (default: Comment)
NvimJupyterOutputText  — stdout/result output (default: String)
NvimJupyterOutputError — error output (default: ErrorMsg)
NvimJupyterOutputCount — execution count [42] (default: Number)
NvimJupyterRunning     — [*] busy indicator (default: WarningMsg)
```

## Key Flows

### Kernel startup

1. Lua reads `metadata.kernelspec.name` from decoded `.ipynb` (e.g. `"python3"`).
2. If daemon not running: `vim.fn.jobstart({bin_path}, {on_stdout=..., on_exit=..., stdout_buffered=false})`.
3. Lua generates `kernel_id` (uuid4), sends `start_kernel` with `kernel_name` and `cwd`.
4. Rust runs `jupyter kernelspec list --json`, parses output to find kernel install path.
5. If kernel not found: Rust sends `kernels_list` → Lua shows `vim.ui.select` picker with available kernel names → user picks → Lua sends `start_kernel` with chosen name.
6. Rust generates connection file at `~/.local/share/nvim-jupyter/<kernel_id>.json` with random ports (49152–65535) and a random 32-byte HMAC key (hex-encoded).
7. Rust spawns kernel: `python -m ipykernel_launcher -f <connection_file>` (or equivalent for other kernels).
8. Rust connects ZMQ sockets (DEALER for shell/control, SUB for iopub, REQ for heartbeat).
9. Rust sends heartbeat ping, waits for pong (timeout 10s). On success: sends `kernel_ready`.
10. Lua receives `kernel_ready` → sets `vim.b.jupyter_kernel_status = "idle"`.

### Cell execution (`<S-CR>` — execute + advance)

1. Lua identifies current cell: walks `ns_cells` marks sorted by line, finds mark whose range contains `vim.api.nvim_win_get_cursor()[1]`.
2. Extracts source: buffer lines from this mark's line to the next mark's line (exclusive), joined with `\n`.
3. Generates `msg_id` (uuid4). Clears `ns_output` extmarks for this cell. Adds `[*]` running indicator.
4. Sets `vim.b.jupyter_kernel_status = "busy"`.
5. Sends `{"cmd":"execute", "kernel_id":..., "msg_id":..., "code":...}`.
6. Rust sends `execute_request` on shell socket (DEALER). Concurrently reads iopub socket (SUB, subscribed to `""`).
7. Each `stream`, `execute_result`, `execute_error` message from iopub is forwarded immediately as an event to stdout.
8. Lua `on_stdout` receives events → appends virtual text lines to `ns_output` for this cell in real time.
9. Rust receives `execute_reply` on shell socket → sends `execute_done`.
10. Lua receives `execute_done`: updates `[42]` execution count, removes `[*]` indicator, sets status `idle`.
11. **Advance:** moves cursor to first line of next cell. If current cell is the last cell, creates a new empty code cell below and moves cursor there.

### `<C-CR>` — execute in place

Same as above steps 1–10, skip step 11 (no cursor movement).

### `<M-CR>` — execute + insert new cell below

Same as above steps 1–10, then insert a new empty code cell immediately below the current cell and move cursor there (regardless of whether current cell is last).

### Restart kernel (`:JupyterRestartKernel`)

1. Lua sends `{"cmd": "restart_kernel", "kernel_id": "..."}`.
2. Rust sends `shutdown_request` with `restart=true` on the control socket and waits for `shutdown_reply`.
3. Rust kills the kernel process (SIGTERM, then SIGKILL after 3s if still alive).
4. Rust respawns a new kernel process using the same connection file (same ports, same HMAC key).
5. Rust reconnects ZMQ sockets, sends heartbeat, waits for pong.
6. Rust sends `kernel_ready` on success.
7. Lua receives `kernel_ready` → clears all `ns_output` virtual text for all cells in this buffer, sets status `idle`.

## Error Handling

| Situation | Behavior |
|-----------|----------|
| Kernel not found at startup | `kernels_list` event → `vim.ui.select` picker. If zero kernels: `vim.notify` with install hint. |
| Kernel process dies unexpectedly | Heartbeat detects 3 missed pongs → `kernel_died`. Lua sets status `dead`, notifies user, offers `:JupyterRestartKernel`. |
| Execute on dead kernel | Lua checks `vim.b.jupyter_kernel_status` before sending — shows error immediately, no IPC call. |
| Python traceback | `execute_error` event rendered as red virtual text (first line of traceback). Full traceback via `:JupyterShowOutput`. |
| Interrupt | `:JupyterInterrupt` → Rust sends `interrupt_request` on control socket → kernel receives SIGINT. |
| Daemon crash | `on_exit` → all open notebook buffers marked dead → user notified. Next execute auto-restarts daemon. |
| Malformed `.ipynb` | `vim.json.decode` failure → `vim.notify` with path and error, abort open. No silent corruption. |
| Cell boundary merge on save | Two `ns_cells` marks at same line → cells merged with `vim.notify` warning. |

## Keymaps

**Requires kitty-protocol terminal** (Ghostty, kitty, wezterm, alacritty). gnome-terminal users use `:Jupyter*` commands directly.

| Key | Action | Command equivalent |
|-----|--------|--------------------|
| `<S-CR>` | Execute cell + advance/create next | `:JupyterExecuteAndAdvance` |
| `<C-CR>` | Execute cell in place | `:JupyterExecute` |
| `<M-CR>` | Execute cell + insert new below | `:JupyterExecuteAndInsert` |
| `]c` | Next cell | `:JupyterNextCell` |
| `[c` | Previous cell | `:JupyterPrevCell` |

All other operations are `:Jupyter*` commands only — no default `<leader>` bindings. Users add their own in their Neovim config.

All five default keymaps are applied buffer-local (`.ipynb` buffers only) and are overridable via `setup({keymaps = false})` or `setup({keymaps = {execute = "<C-CR>", ...}})`.

### Full command list

```
:JupyterExecute             execute current cell in place
:JupyterExecuteAndAdvance   execute + move to/create next cell
:JupyterExecuteAndInsert    execute + insert new cell below
:JupyterExecuteAll          execute all cells top to bottom
:JupyterNextCell            move cursor to next cell
:JupyterPrevCell            move cursor to previous cell
:JupyterAddCellBelow        insert new code cell below current
:JupyterAddCellAbove        insert new code cell above current
:JupyterDeleteCell          delete current cell (no undo via u — use :JupyterUndoDeleteCell)
:JupyterUndoDeleteCell      restore last deleted cell
:JupyterChangeCellType      toggle current cell between code and markdown
:JupyterKernel              show kernel picker (swap kernel for this buffer)
:JupyterRestartKernel       restart kernel for this buffer
:JupyterInterrupt           send interrupt (SIGINT) to kernel
:JupyterKernelStatus        print current kernel status
:JupyterShowOutput          show full output for current cell in a scratch split
:JupyterBuild               build the nvim-jupyter Rust binary
```

### Vim conflict avoidance

No bare letter or common Vim motion keys are intercepted. Specifically:
- `dd` deletes a buffer line (correct Vim behavior) — never deletes a cell
- `u` undoes text edits — never undoes cell-level operations (`:JupyterUndoDeleteCell` is the explicit command)
- `j` / `k` move by line — cell navigation uses `]c` / `[c`
- `a`, `b`, `o`, `x`, `y`, `m`, `s` are untouched

## Testing

### Rust unit tests (`cargo test`)

- `protocol.rs` — round-trip serialize/deserialize every `Command` and `Event` variant
- `wire.rs` — HMAC-SHA256 signing/verification against known Jupyter wire protocol test vectors
- `kernel.rs` — connection file generation: valid JSON, ports in range 49152–65535, unique HMAC key each call

### Rust integration test (`cargo test --features integration`)

Located at `tests/kernel_integration.rs`. Requires Python + ipykernel installed.
- Starts a real `python3` kernel
- Sends `execute_request` with `1+1`
- Asserts `execute_result` text is `"2"` and `execute_done` status is `"ok"`
- Gated behind `#[cfg(feature = "integration")]` — not run in standard CI

- `test/cells_spec.lua` — extmark boundary detection: given cursor line, assert correct cell identified; boundary merge on delete

`test/smoke.sh` — builds binary, launches Neovim with `test/fixtures/simple.ipynb`, prints checklist for human verification:
1. Cells display with separators
2. `<S-CR>` executes and advances
3. Output appears as virtual text
4. `:JupyterRestartKernel` clears status and reconnects
5. `:w` saves without corrupting the `.ipynb`

## Implementation Notes for External Agents

- The Rust binary is at `bin/nvim-jupyter` (built by `build.sh`). Lua discovers it relative to the plugin root using `debug.getinfo` — same pattern as nvim-gfx.
- The Jupyter wire protocol requires HMAC-SHA256 message signing. The signature covers the concatenation of: header, parent_header, metadata, content (all as JSON bytes). See [Jupyter Messaging Docs](https://jupyter-client.readthedocs.io/en/stable/messaging.html).
- ZMQ socket types: shell = DEALER, iopub = SUB (subscribe to `""`), control = DEALER, heartbeat = REQ. Do not use PUSH/PULL — the Jupyter protocol specifies DEALER/ROUTER for shell.
- Connection file format is documented in the Jupyter kernel spec: `~/.local/share/nvim-jupyter/<kernel_id>.json` with fields `shell_port`, `iopub_port`, `stdin_port`, `control_port`, `hb_port`, `ip`, `key`, `transport`, `signature_scheme`, `kernel_name`.
- `vim.b[bufnr].jupyter_kernel_status` is the canonical kernel state readable by statusline plugins (e.g. lualine).
- The nvim-gfx sibling project at `/home/robertcowher/rustprojects/neovim-graphics-viewer` demonstrates the full jobstart/chansend/on_stdout/on_exit Lua pattern. Read `lua/nvim-gfx/viewer.lua` before implementing `daemon.lua`.
